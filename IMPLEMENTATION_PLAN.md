# comiket-tsp — Implementation Plan

A Rust CLI that plans an efficient walking route around Comiket circles as a TSP.
Two decoupled stages: **layout generation** and **route search**.

---

## 1. Goal & scope

- Input A (reused): a *block table* describing each island's position and numbering —
  `block_layouts.csv` (generated by `tools/layout/`, or hand-authored).
- Input B (changes often): the circles to visit — `want_list.csv`, authored in Excel.
- Output: an ordered itinerary minimizing total *perceived walking cost*.
- Size: up to ~200 wanted circles. Whole-venue layout is ~100–150 islands.
- Quality target: near-optimal (within a few % of optimal) in well under a second
  for the solve stage. Exactness is not required.

Non-goals (for now): time windows / sell-out priorities, one-way aisle asymmetry,
real-time crowd data. The design leaves room for these (see § 9).

---

## 2. Domain model

### 2.1 Space

A space is one circle's half-table. Notation decomposes into:

```
building : East | West | South     // 東 / 西 / 南
hall     : u8                       // 4, 5, 6, 7, 8, 1, 2 ...
block    : String                   // "ア", "あ", "A", "t" ... (1 letter, multi-byte ok)
number   : u16                      // serpentine index within the island
side     : A | B                    // left/right half of one table
```

`SpaceId` = `(building, hall, block, number, side)`. It must round-trip through a
canonical string (e.g. `E4-ア-31a`) for CSV matching against the want-list.

### 2.2 Block (one island)

Authored once per event. Few fields per row:

```rust
struct Block {
    id:           String,        // "E4-ア"
    building:     Building,
    hall:         u8,
    anchor:       (f64, f64),     // global meters: the n=1, face0, side-a desk
    along:        Dir,            // unit dir of increasing depth (N/S/E/W)
    cross:        Dir,            // unit dir from face0 to face1
    n_max:        u16,            // highest number in this island
    face0_len:    Option<u16>,    // override split for irregular islands (default n_max/2)
    pitch:        f64,            // desk spacing along the island, ≈ 1.0 m
    island_width: f64,           // distance between the two faces, ≈ 1.5 m
}
```

### 2.3 Serpentine coordinate derivation

Numbers run up one face and back down the other (see chat diagram). For number `n`:

```
half  = face0_len.unwrap_or(n_max / 2)
if n <= half {
    face  = 0
    depth = n
} else {
    face  = 1
    depth = n_max - n + 1
}
along_m = (depth - 1) as f64 * pitch
cross_m = face as f64 * island_width + side_offset(side)   // side_offset(a)=0, (b)=~0.45
global  = anchor + along_m * along.unit() + cross_m * cross.unit()
```

Edge cases to handle:
- Odd `n_max` / unequal faces → use `face0_len` override.
- Island-head ("お誕生日席") seats and `2日目はありません` blanks: ignore for 概算;
  they don't shift coordinates materially. Document the assumption.
- `a`/`b` are the same face, ~0.45 m apart laterally.

## 3. Distance model

Hybrid of coordinate distance (case 2) + tiered structural penalties (case 1),
then a nonlinear squash so long legs and building/hall crossings dominate.

```rust
fn perceived(a: &Space, b: &Space) -> f64 {
    // Same-island legs use an around-the-nearer-end walk (you can't cut through the
    // tables); everything else uses global Manhattan. See `intra_island` below.
    let mut d = if same_island(a, b) { intra_island(a, b) }
                else                 { manhattan(a.coord, b.coord) };  // base walking meters
    d += if a.building != b.building { 250.0 }         // cross-building: galleria/security
         else if a.hall  != b.hall  { 40.0 }            // cross-hall
         else if a.block != b.block { 5.0 }             // adjacent island (one aisle)
         else { 0.0 };                                  // same island
    d.powf(GAMMA)                                       // GAMMA = 1.25
}

// Within one island: same face → straight along the aisle; opposite faces → round
// whichever island end is nearer (L = island_len). This fixes Manhattan's
// underestimate of the opposite-face walk without overestimating near-end pairs.
fn intra_island(a: &Space, b: &Space) -> f64 {
    let dcross = (a.cross - b.cross).abs();
    if a.face == b.face { (a.along - b.along).abs() + dcross }
    else {
        let (near, far) = (a.along + b.along, (L - a.along) + (L - b.along));
        near.min(far) + dcross
    }
}
```

Effect: a building jump (~+250) becomes ~990 after the squash, so the solver
naturally clears one hall/building before moving on — the real Comiket strategy.

Notes / constants are tunable; expose `GAMMA` and the penalties as `solve` flags
(`--gamma`, `--pen-building`, …) with the defaults above. `--pen-corridor` (default
`1.0`) adds a small cost to crossing an island's faces through a 細通路 (minor aisle) or
the far turnaround, but not through a 太通路 (中央通路 / 大通路) — so the search prefers the
wider, cheaper aisles (太通路の方が安い). The crossings that `intra_island` rounds carry a
`major`/`minor` tag for this; see §11.1.

Caveats to encode as comments/tests:
- `powf` breaks the triangle inequality → metric-based bounds (Christofides) don't
  apply. Fine: 2-opt / Or-opt / SA / ILS work on arbitrary cost matrices.
- Same-island opposite-face walks are modelled exactly by `intra_island` (round the
  nearer island end), so Manhattan's underestimate there does not apply. Manhattan is
  still used for *cross*-island legs (it ignores obstacles between islands — acceptable
  for 概算).
- The matrix is symmetric for now. Keep the type asymmetric-ready (`d[i][j]`) so a
  future one-way-aisle model is a data change, not a code change.

## 4. CLI design (clap, derive)

```
comiket-tsp gen-layout --blocks <csv> --out <json> [--event <name>]
comiket-tsp solve      --spaces <json> --want <csv> --out <csv>
                       [--seed 42] [--restarts 16] [--time-ms 800] [--max-iters 100000]
                       [--start <SpaceId>] [--closed]
                       [--gamma 1.25] [--pen-building 250] [--pen-hall 40] [--pen-block 5]
                       [--pen-corridor 1.0]
```

- `--start` fixes the first stop; it must be a space present in the want-list. Default:
  solver picks the best start.
- `--closed` returns to the start (round trip). Default: open path.
- `--max-iters` caps ILS iterations per restart and is the deterministic stopping rule.
  `--time-ms 0` disables the wall-clock limit, making runs byte-identical for a seed.

## 6. Algorithm (solve stage)

1. **Build matrix.** Look up each wanted space's coords from the layout; warn on
   any want-list entry not found, then `perceived()` over all pairs → `N×N` `f64`
   (`N ≤ 200`, ~320 KB — precompute once, reuse everywhere).
2. **Construct.** Nearest-neighbor from the start. If `--start` is unset, try every
   node as a start (or a sampled subset) and keep the best NN tour.
3. **Local search.** 2-opt (first-improvement, segment-reverse) interleaved with
   Or-opt (relocate segments of length 1–3). Loop until no improving move — a
   2-opt/Or-opt local optimum. Respect fixed endpoints in open-path mode (don't
   move index 0; don't wrap).
4. **Iterated local search (primary metaheuristic).** Perturb the local optimum
   with a **double-bridge** (4-opt) kick, re-run step 3, accept if better (or with
   an SA-style probability). Repeat until `--time-ms` or an iteration cap. ILS +
   double-bridge is simple and strong for this size.
   - Optional alternative: classic simulated annealing over the same neighborhood,
     geometric cooling. Keep behind the same trait so either can be selected.
5. **Parallel restarts.** Run `--restarts` independent ILS chains via `rayon`,
   each with a deterministic child seed; keep the global best.
6. **Output.** Emit `route.csv`: visit order, space id, circle name, leg cost,
   cumulative cost; print total perceived cost and a short human itinerary
   (grouped by building/hall).

Tour representation: `Vec<usize>` of node indices. Cache tour cost and update it
incrementally on each accepted move (don't recompute from scratch).

Open path vs cycle: implement both. Open path = fixed first node, free end, no
wrap-around edge. Cycle = `--closed`, includes the return edge.

## 7. Data formats

### block_layouts.csv
```
id,building,hall,anchor_x,anchor_y,along,cross,n_max,face0_len,pitch,island_width,kind,cluster,along_deg,cross_deg,crossings,number_base
E6-ウ,East,6,260,0,S,E,66,,1,1.5,island,E4-6,,,16@major;24.5@minor;8.5@minor,0
E4-ア,East,4,295,8,W,S,51,,5.686,1.5,wall,E4-6,,,,22
E8-W,East,8,80,-10,W,S,68,,1,1.5,island,E7-8,215,305,33@minor,0
```
`along`/`cross` ∈ {N,S,E,W}. Empty `face0_len` ⇒ `n_max/2`. The last six columns are
optional (defaulted) so older files still parse — see §11. (Was `block_layout.csv`.)
`crossings` is `;`-separated `along@kind` where `kind` ∈ {`major`,`minor`} (太通路/細通路);
a bare `along` (no `@kind`) parses as `minor`, so legacy files are unchanged. `number_base`
(default `0`) offsets serpentine numbers so a 壁サー split into several wall rows keeps
continuous numbering (the `E4-ア` row above is its 上壁 segment, numbers 23–73); each wall
segment uses a fractional `pitch` to spread its circles 等間隔 over the wall's length.

### want_list.csv (from Excel)
```
space,name
E4-ア-31a,さくらサークル
W1-あ-12b,Example Circle
```
`space` is the canonical id; `name` is free text for the itinerary.

### spaces.json (artifact, gen-layout → solve)
```json
{
  "event": "C107",
  "spaces": [
    { "id": "E4-ア-31a", "building": "East", "hall": 4,
      "block": "ア", "number": 31, "side": "A",
      "x": 12.0, "y": 33.0,
      "along": 30.0, "cross": 0.0, "face": 0, "island_len": 47.0 }
  ]
}
```

`x`/`y` are global metres (cross-island Manhattan). `along`/`cross` are local island
metres, `face` ∈ {0,1}, and `island_len` (= `(max_face_len - 1) * pitch`) the island's
far-end along-position — all four feed the same-island `intra_island` model, so `solve`
never re-reads `block_layouts.csv`. Each space also carries `cluster` and `crossings`
(see §11). `event` is optional (omitted when not given).

### hall_distances.csv (square matrix, optional input to `solve`)
```
cluster,E1-3,E4-6,E7-8,W1-2,S1,S2
E4-6,120,0,150,450,600,660
...
```
First column = row label, header row = column labels, cells = perceived metres. Symmetric
(unfilled lower triangle is fine); blank ⇒ unknown (falls back to the penalty model).

## 10. References (algorithms & Rust)

TSP in Rust:
- `ibn_battuta` (crates.io / github BIRSAx2/ibn-battuta) — full library: NN, 2-opt,
  3-opt, Lin-Kernighan, SA, GA, ACO, rayon, TSPLIB parser. Best reference for move
  implementations; optional dependency if hand-rolling stalls.
- `localsearch` (lib.rs) — generic local-search framework via an `OptModel` trait,
  rayon-parallel. Use if you'd rather plug into a framework than hand-roll.
- `aprender-tsp` — CLI + library with seeded determinism and TSPLIB/CSV input; good
  reference for CLI ergonomics and benchmarking against known optima.
- `stoksc/tsp-rs` — small, readable 2-opt(+3-opt) reference.
- `onkolahmet/TSP` — writeup of 2-opt with fast/guided local search.

Why hand-roll the core here: the cost matrix is non-metric (the `powf` squash) and
we need fixed-start open paths — both are awkward to express through a generic TSP
crate, and the move code is small. Crib details from the above; own the orchestration.

Benchmarks: TSPLIB instances (berlin52, eil51, att48) for validating the solver
against published optima before trusting it on venue data.

Concepts to implement (keywords for lookup): nearest-neighbor construction, 2-opt
segment reversal, Or-opt (segment relocation), double-bridge 4-opt perturbation,
iterated local search (ILS), simulated annealing with geometric cooling.

## 11. Real-venue extensions (壁・通路・コの字・斜め・ホール間距離)

Built on top of P0–P8 so the model matches the actual C107 map, and so a Python tool can
author layouts from per-event constants. All new CSV columns default, so legacy files and
omitting `--hall-distances` reproduce the original behaviour.

### 11.1 Block kinds & geometry (`src/layout.rs`)
- `BlockKind { Island, Wall }`. **Wall (壁サークル)** = single face; numbers run linearly
  up face 0 (no serpentine wrap). `kind` column, default `island`. A 壁サー that folds
  around several walls is authored as one wall row *per straight segment* (`WallSegment`
  in `config.py`), sharing the block id; `number_base` keeps numbering continuous and a
  fractional `pitch` (= span / count) spreads each segment 等間隔. The 東7 diagonal wall
  uses `angle_deg` like the diagonal islands.
- **Diagonal halls (東7/8):** optional `along_deg`/`cross_deg` (degrees CCW from East)
  override the cardinal `along`/`cross`, so an island can sit at any angle.
- **コの字 (西1/2):** no special case — the U is just several blocks/rows with different
  cardinal orientations (left wing depth S, top E, etc.). The generator emits them as
  ordinary rows.
- **Corridors splitting a prefix (同一接頭辞を区切る通路):** `crossings` = `;`-separated
  `along@kind` entries (metres + 太通路/細通路) where the two faces can be crossed. The
  island ends are always implied (near end = `major` 大通路, far turnaround = `minor`); mid
  values are the authored corridors. Stored on every `Space`; `intra_island` rounds the
  least-cost crossing, charging `--pen-corridor` for `minor` ones (see §3). Corridors are
  authored in `tools/layout/config.py` as `Corridor(face0_after, face1_after, kind)`
  number-breaks per the per-pattern spec (`island_implemantation.txt`), so the Excel map,
  the CSV, and the solver agree on every aisle.

### 11.2 Hall clusters & distance matrix (`src/distance.rs`, `src/io.rs`)
- Each space carries a `cluster` id (`E1-3`, `E4-6`, `E7-8`, `W1-2`, `S1`, `S2`); the
  default map lives in `default_cluster(building, hall)`, overridable per row.
- `HallDistances` (loaded from `hall_distances.csv`) gives door-to-door cluster distances.
- `perceived(a, b, params, hall)`:
  - same island → `intra_island` (now rounds the nearest of {ends ∪ corridors}).
  - `hall` present & different cluster → **matrix value is authoritative** (replaces the
    building/hall penalties), then `^gamma`.
  - else (same cluster, different island) → Manhattan + hall/block penalty, then `^gamma`.
  - `hall = None` → original penalty-only model (legacy inputs unchanged).
- `solve` gains `--hall-distances <csv>` (optional).

### 11.3 Python layout generator (`tools/layout/`)
- `config.py` — per-event constants as dataclasses (`Cluster`, `Island`, `Row`, `Hall`,
  `EventConfig`) plus the `C107` instance and an `EVENTS` registry. **This is the only
  file to edit for C108+.**
- `model.py` — the placement "law": rows of islands along a prefix axis, serpentine/wall
  expansion (kept in lock-step with `src/layout.rs`), `BlockRecord` generation.
- `csvout.py` — writes `block_layouts.csv` and the square `hall_distances.csv`.
- `excel.py` — `layout_<event>.xlsx`: per-hall detail sheets (1 cell = 1 circle), an
  island overview (1 cell = 1 island), the distance matrix, and a block table.
- `generate.py` — `python -m tools.layout.generate --event C107`.

### 11.4 Known approximations
Within a hall, cross-island distance is still Manhattan; the open mouth of the コの字 and
the diagonal band are placed with correct geometry but not routed around. A per-hall
"spine" graph would fix this and is the natural next hook (slots in at `perceived`'s
same-cluster branch). The "太通路 is cheaper" rule is modelled only *inside* an island
(face crossings via `--pen-corridor`); the wider aisles *between* islands (縦大通路 / 横大
通路) widen the Manhattan gap geometrically but are not yet discounted — that also wants
the spine graph.

Per-island serpentine + corridors now follow `island_implemantation.txt` for 東4/5/6
(and 東1/2/3 by the same five patterns), 東7, 南1/2, and 西1/2 (top bands + four-block
コの字 wings). **Walls (壁サー)** are authored as folded multi-segment runs
(`WallSegment` per straight wall): each segment carries its own number range, direction
(incl. the 東7 diagonal via `angle_deg`), and a physical `span` over which its circles
are spread 等間隔; numbering stays continuous across segments via `number_base`, and
skipped ranges (e.g. め/あ 62–65) simply have no segment. One structure stays
approximate: the **東8** islands (two side-by-side serpentine blocks each) are modelled
as one combined serpentine with an auto mid corridor — a small fraction of the spaces
that only affects its own local geometry.
